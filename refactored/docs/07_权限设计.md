# 07 权限设计

## 1. 概述

本文档定义图书管理系统的权限控制体系，采用RBAC（Role-Based Access Control）模型，实现基于角色的细粒度权限管理。

---

## 2. RBAC权限模型

### 2.1 模型架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           RBAC权限模型架构                                   │

┌─────────────┐       ┌─────────────┐       ┌─────────────┐       ┌─────────────┐
```

### 2.2 核心概念

| 概念 | 说明 | 示例 |
|------|------|------|
| **用户(User)** | 系统的使用者 | 张三、李四 |
| **角色(Role)** | 一组权限的集合 | admin、librarian、reader |
| **权限(Permission)** | 对资源的操作许可 | book:create、user:read |
| **资源(Resource)** | 被保护的对象 | 图书、用户、借阅记录 |
| **操作(Action)** | 对资源的具体操作 | create、read、update、delete |

## 3. 角色定义

### 3.1 角色层级

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           角色层级结构                                       │
└─────────────────────────────────────────────────────────────────────────────┘

                    ┌─────────────┐
                    │    admin    │
                    │  (系统管理员) │
                    └──────┬──────┘
                           │
           ┌───────────────┼───────────────┐
           │               │               │
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │  librarian  │ │   reader    │ │   guest     │
    │ (图书管理员)  │ │   (读者)     │ │  (访客)      │
    └─────────────┘ └─────────────┘ └─────────────┘

权限继承关系:
- admin 继承所有角色的权限
- librarian 拥有 reader 的所有权限 + 图书管理权限
- reader 拥有 guest 的所有权限 + 借阅权限
```

### 3.2 角色详细定义

#### 3.2.1 角色枚举与继承

**设计思路**: 使用 `Role` 枚举定义四种角色，通过 `ROLE_HIERARCHY` 字典建立继承关系。`get_role_permissions()` 递归获取角色及其继承角色的所有权限，`has_permission()` 提供快捷权限检查。

**角色定义**:
- `Role.ADMIN = "admin"` -- 系统管理员，level=1
- `Role.LIBRARIAN = "librarian"` -- 图书管理员，level=2
- `Role.READER = "reader"` -- 读者，level=3
- `Role.GUEST = "guest"` -- 访客（未登录），level=4

**继承关系**:
- admin -> [librarian, reader, guest]
- librarian -> [reader, guest]
- reader -> [guest]
- guest -> []

> 实现文件: `app/core/permissions.py`

#### 3.2.2 角色权限范围

| 角色 | 权限范围 | 典型用户 |
|------|----------|----------|
| admin | 系统所有功能 | 系统管理员、技术负责人 |
| librarian | 图书管理、借阅管理、用户查询 | 图书馆工作人员 |
| reader | 图书查询、借阅、个人中心 | 注册用户 |
| guest | 图书浏览 | 未登录用户 |

### 4.2 权限定义

**设计思路**: 使用 `Permission(str, Enum)` 枚举定义所有权限，格式为 `{资源}:{操作}`。通过 `ROLE_PERMISSIONS` 字典建立角色与权限的映射关系，admin 拥有所有权限，其他角色按需分配。

**权限分类**:
- 系统管理: `system:config`, `role:manage`, `permission:manage`, `log:view`, `backup:manage`
- 用户管理: `user:create`, `user:read`, `user:update`, `user:delete`, `user:freeze`, `user:blacklist`, `user:read:all`, `user:update:self`
- 图书管理: `book:create`, `book:read`, `book:update`, `book:delete`, `book:import`, `book:mark_lost`
- 分类管理: `category:create`, `category:read`, `category:update`, `category:delete`
- 借阅管理: `borrow:create`, `borrow:return`, `borrow:renew`, `borrow:read`, `borrow:read:all`, `borrow:proxy`
- 逾期管理: `overdue:read`, `overdue:read:all`, `fine:pay`, `fine:waive`
- 统计报表: `stats:view`, `stats:view:all`, `report:export`

> 实现文件: `app/core/permissions.py`

## 5. 资源定义

### 5.1 资源分类

**设计思路**: 使用 `ResourceType` 枚举定义资源类型，`ResourceAction` 枚举定义操作类型。`RESOURCES` 字典描述每种资源支持的操作及数据权限判断字段（`ownership_field`）。

**资源类型**: user, user_profile, book, category, borrow_record, overdue_record, system, role, permission, log, report

**操作类型**: create, read, update, delete, list, export, import, borrow, return, renew, freeze, unfreeze, blacklist, unblacklist, mark_lost, pay_fine, waive_fine

> 实现文件: `app/core/resources.py`

### 5.2 数据权限

**设计思路**: `DataPermissionChecker` 根据用户角色返回不同的数据范围过滤条件:
- **admin**: 可访问所有数据，无过滤
- **librarian**: 可访问业务数据（图书、分类），用户数据需脱敏
- **reader**: 只能访问自己的数据（通过 `ownership_field` 匹配 `user_id`）

> 实现文件: `app/core/data_permission.py`

## 6. 前端权限控制

### 6.1 菜单权限

**设计思路**: 定义 `MenuItem` 接口，每个菜单项绑定所需的 `permissions` 数组。`hasPermission()` 检查用户是否拥有所需权限（满足任一即可），`filterMenus()` 递归过滤菜单树，移除用户无权访问的菜单项。

**菜单结构**:
- 首页 (`/dashboard`) -- 所有登录用户
- 图书管理 (`/books`) -- 列表、新增、编辑、批量导入
- 分类管理 (`/categories`)
- 借阅管理 (`/borrow`) -- 借阅记录、归还图书、逾期管理
- 用户管理 (`/users`) -- 用户列表、黑名单
- 统计报表 (`/stats`) -- 借阅统计、综合统计
- 系统管理 (`/system`) -- 系统配置、操作日志

> 实现文件: `web/src/permission/menu.ts`

### 6.2 按钮权限

**设计思路**: 封装 `PermissionButton` 组件，接收 `permission` 属性（单个或数组）。有权限时渲染插槽内容，无权限时可选择显示禁用态（`showDisabled`）或完全隐藏。

> 实现文件: `web/src/components/PermissionButton.vue`

### 6.3 路由权限守卫

**设计思路**: 在 `router.beforeEach` 中实现三级检查:
1. 白名单路由直接放行（login、register、404）
2. 未登录用户重定向到登录页
3. 已登录用户检查路由 `meta.permissions`，无权则跳转 403

> 实现文件: `web/src/router/guard.ts`

## 7. 后端权限校验

### 7.1 权限装饰器

**设计思路**: 提供 `require_permissions` 和 `require_role` 两个装饰器，从请求中提取当前用户并校验权限。支持 `operator="any"`（满足任一）和 `operator="all"`（满足全部）两种模式。还提供 `require_admin`、`require_librarian`、`require_login` 快捷装饰器。

> 实现文件: `app/core/decorators.py`

### 7.2 权限依赖注入

**设计思路**: 使用 FastAPI 的 `Depends` 机制实现权限校验:
- `get_current_user`: 从 JWT Token 中解析用户信息并查询数据库
- `PermissionChecker`: 依赖类，检查用户角色是否拥有所需权限
- `check_permission(*permissions)`: 创建权限检查依赖的工厂函数
- `check_ownership(resource_owner_id)`: 检查资源所有权，admin/librarian 可访问所有资源

> 实现文件: `app/core/dependencies.py`

### 7.3 中间件权限校验

**设计思路**: `PermissionMiddleware` 作为全局中间件，维护公开路由白名单和路由权限映射表（`PATH_PERMISSIONS`）。对非公开路由，验证 JWT Token 后根据路径和方法匹配所需权限，支持精确匹配和前缀匹配。

**公开路由**: `/api/v1/auth/login`, `/api/v1/auth/register`, `/api/v1/auth/refresh`, `/api/v1/health`, `/docs`, `/openapi.json`

> 实现文件: `app/middleware/permission_middleware.py`

### 7.4 API权限使用示例

**设计思路**: API 端点通过 `Depends(check_permission(Permission.XXX))` 声明所需权限。对于涉及所有权的操作（如查看用户详情），先检查是否为本人，非本人则需要 `user:read:all` 权限。

**关键端点权限要求**:
- `GET /books` -- 所有登录用户
- `POST /books` -- `book:create` (admin, librarian)
- `PUT /books/{id}` -- `book:update` (admin, librarian)
- `DELETE /books/{id}` -- `book:delete` (仅admin)
- `GET /users` -- `user:read:all` (admin, librarian)
- `POST /users/{id}/freeze` -- `user:freeze` (admin, librarian)
- `POST /users/{id}/blacklist` -- `user:blacklist` (admin, librarian)

## 8. 权限缓存优化

**设计思路**: `PermissionCache` 使用 Redis 缓存角色权限数据，避免每次请求都递归计算继承权限。缓存 key 格式为 `permissions:role:{role}`，TTL 为 1 小时。当角色权限变更时，调用 `invalidate_role_cache()` 或 `invalidate_all_cache()` 使缓存失效。

> 实现文件: `app/core/permission_cache.py`
