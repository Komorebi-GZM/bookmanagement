## 1. 测试策略

### 1.1 测试金字塔

```
                    ▲
                   /  \
                  / E2E \          端到端测试 (5%)
                 / 测试  \         - 用户场景验证
                /─────────\
               /  集成测试  \       集成测试 (15%)
              /   (API/DB)  \      - 服务间交互
             /───────────────\
            /    单元测试      \     单元测试 (80%)
           /   (业务逻辑/工具)   \    - 函数/方法验证
          /─────────────────────\
```

### 1.2 测试分层策略

| 测试类型 | 比例 | 工具 | 执行时机 | 目标 |
|----------|------|------|----------|------|
| **单元测试** | 80% | pytest | 每次提交 | 验证最小代码单元 |
| **集成测试** | 15% | pytest + TestContainers | 合并前 | 验证组件交互 |
| **E2E测试** | 5% | Playwright | 发布前 | 验证完整用户流程 |

### 1.3 测试环境架构

```
┌─────────────────────────────────────────────────────────────┐
│                      测试环境矩阵                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │   单元测试    │  │   集成测试    │  │   E2E测试    │      │
│  │   (本地)      │  │   (CI容器)    │  │   (预发布)    │      │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘      │
│         │                 │                 │              │
│         ▼                 ▼                 ▼              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │  SQLite内存   │  │  PostgreSQL  │  │  PostgreSQL  │      │
│  │  (Mock外部)   │  │  (TestCont.) │  │  (生产数据)  │      │
│  └──────────────┘  └──────────────┘  └──────────────┘      │
│                                                             │
│  执行速度: 毫秒级      秒级              分钟级              │
```

核心依赖: `pytest`, `pytest-asyncio`, `pytest-cov`, `pytest-xdist`

| 类别 | 工具 | 用途 |
|------|------|------|

## 3. 测试用例设计思路

### 3.1 测试用例设计原则

```
┌─────────────────────────────────────────────────────────────┐
│                    AAA模式 (Arrange-Act-Assert)              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Arrange (准备)                                             │
│  ├── 初始化测试数据                                         │
│  ├── 设置Mock对象                                           │
│  └── 配置测试环境                                           │
│                                                             │
│  Act (执行)                                                 │
│  ├── 调用被测函数/方法                                      │
│  └── 触发被测行为                                           │
│                                                             │
│  Assert (断言)                                              │
│  ├── 验证返回值                                             │
│  ├── 验证状态变更                                           │
│  └── 验证副作用 (Mock调用)                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 单元测试设计要点

- 使用 `factory-boy` 创建测试数据工厂，避免硬编码测试数据
- 使用 `AsyncMock` 模拟异步数据库操作和外部服务
- 每个测试方法只验证一个行为点，保持测试独立性
- 异常测试使用 `pytest.raises` 验证异常类型和消息

### 3.3 边界值测试设计要点

- 使用 `@pytest.mark.parametrize` 进行参数化测试
- 密码验证: 覆盖最小/最大长度、缺少各类字符、弱密码、空格等边界
- ISBN验证: 覆盖ISBN-10/ISBN-13有效格式、长度错误、校验位错误、空值等

### 3.4 集成测试设计要点

- 使用 `httpx.AsyncClient` 创建测试客户端，不启动真实服务器
- 测试流程: 创建认证头 -> 发送请求 -> 验证状态码和响应体
- 覆盖正常流程和异常流程(重复ISBN、资源不存在等)

### 3.5 测试数据工厂设计思路

使用 `factory-boy` + SQLAlchemy 为每个模型创建工厂类:
- **CategoryFactory**: 分类数据，自动生成code和排序
- **UserFactory**: 用户数据，支持角色和状态配置
- **BookFactory**: 图书数据，使用 `SubFactory` 关联分类，自动计算可借数量
- **BorrowFactory**: 借阅记录，关联用户和图书，自动生成借阅/应还日期

## 4. 并发测试方案

### 4.1 并发测试场景

```
┌─────────────────────────────────────────────────────────────┐
│                     并发测试场景                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 库存竞争场景                                             │
│     └── 多用户同时借阅最后一本图书                            │
│         └── 预期: 只有1人成功，其余失败                       │
│                                                             │
│  2. 重复提交场景                                             │
│     └── 用户快速双击提交按钮                                  │
│         └── 预期: 只处理一次，幂等性保证                      │
│                                                             │
│  3. 并发修改场景                                             │
│     └── 管理员A和B同时修改同一图书信息                        │
│         └── 预期: 乐观锁防止丢失更新                          │
│                                                             │
│  4. 限流测试场景                                             │
│     └── 突发流量超过API限流阈值                               │
│         └── 预期: 超限请求被优雅拒绝                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 并发测试实现思路

- 使用 `asyncio.gather` 模拟多用户并发请求
- **库存竞争**: 创建1本库存图书，5个用户同时借阅，断言只有1人成功
- **幂等性测试**: 使用 `Idempotency-Key` 请求头，验证重复请求被正确处理
- **乐观锁测试**: 两个并发更新同一记录(version字段)，断言只有1个成功

### 4.3 压力测试方案

使用 Locust 进行压力测试，模拟两类用户行为:

| 用户类型 | 权重 | 行为模拟 |
|----------|------|----------|
| **普通用户** | 90% | 浏览图书(10)、搜索(5)、查看详情(3)、借阅(2)、归还(1) |
| **管理员** | 10% | 添加图书(3)、查看统计(2) |

执行命令: `locust -f tests/load/locustfile.py --host=http://localhost:8000 --users=1000 --spawn-rate=100 --run-time=10m`

## 5. 性能测试指标

### 5.1 性能指标定义

| 指标类别 | 指标名称 | 目标值 | 警告阈值 | 严重阈值 |
|----------|----------|--------|----------|----------|
| **响应时间** | P50 | < 100ms | 200ms | 500ms |
| | P95 | < 300ms | 500ms | 1000ms |
| | P99 | < 500ms | 1000ms | 2000ms |
| **吞吐量** | QPS | > 1000 | 800 | 500 |
| **错误率** | 5xx错误 | < 0.1% | 0.5% | 1% |
| **资源使用** | CPU | < 70% | 80% | 90% |
| | 内存 | < 70% | 80% | 90% |
| | 数据库连接 | < 80% | 90% | 100% |

### 5.2 性能测试方法

- **API性能基准**: 对每个端点执行多次请求，统计P50/P95/P99延迟
- **数据库查询性能**: 测量简单查询(<5ms)、搜索查询(<50ms)、统计查询(<100ms)
- 使用 `@pytest.mark.benchmark` 标记性能测试用例，与常规测试分离执行

## 6. 验收标准

### 6.1 功能验收标准

| 功能模块 | 验收项 | 验收标准 | 验证方法 |
|----------|--------|----------|----------|
| **用户认证** | 登录功能 | 正确凭证登录成功，错误凭证拒绝 | 单元+集成测试 |
| | 注册功能 | 数据校验通过，重复用户拒绝 | 单元+集成测试 |
| | Token刷新 | 过期前可刷新，过期后需重新登录 | 集成测试 |
| **图书管理** | 图书CRUD | 增删改查功能完整 | 单元+集成+E2E |
| | 图书搜索 | 支持多条件组合搜索 | 集成测试 |
| | 库存管理 | 借还书库存正确更新 | 单元+并发测试 |
| **借阅管理** | 借书流程 | 正常借书成功，异常条件拒绝 | 单元+集成+E2E |
| | 还书流程 | 正常还书成功，逾期计算正确 | 单元+集成测试 |
| | 续借功能 | 符合条件可续借 | 集成测试 |
| **权限控制** | 角色权限 | 不同角色访问权限正确 | 集成测试 |
| | 越权防护 | 无权限操作被拒绝 | 安全测试 |

### 6.2 质量门禁配置

```toml
[tool.coverage.run]
source = ["app"]
omit = ["*/tests/*", "*/migrations/*"]

[tool.coverage.report]
fail_under = 80

[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "-v --tb=short --strict-markers --cov=app --cov-report=term-missing --cov-fail-under=80"
markers = [
    "slow: marks tests as slow",
    "integration: marks tests as integration tests",
    "e2e: marks tests as end-to-end tests",
    "benchmark: marks tests as performance benchmarks",
]
```

### 6.3 CI/CD测试流水线

```
push/PR → lint(代码检查) → unit-test(单元测试) → integration-test(集成测试) → e2e-test(E2E测试)
                                                                      ↓
                                                              performance-test(仅main分支)
```

流水线阶段说明:
- **lint**: ruff检查 + black格式化 + mypy类型检查 + bandit安全扫描
- **unit-test**: 运行单元测试，覆盖率上传Codecov
- **integration-test**: 启动PostgreSQL/Redis容器，运行集成测试
- **e2e-test**: 启动完整环境，Playwright执行E2E测试
- **performance-test**: Locust压力测试(100用户/5分钟)，生成HTML报告

### 6.4 测试命名规范

| 层级 | 命名模式 | 示例 |
|------|----------|------|
| 测试文件 | `test_{模块名}.py` | `test_borrow_service.py` |
| 测试类 | `Test{被测类名}` | `TestBorrowService` |
| 测试方法 | `test_{场景}_{预期结果}` | `test_borrow_book_success` |
| Fixture | `{被测对象}_fixture` | `borrow_service_fixture` |
