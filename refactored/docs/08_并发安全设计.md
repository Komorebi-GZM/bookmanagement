# 08 并发安全设计

## 1. 概述

本文档定义图书管理系统的并发安全策略，针对高并发借书/还书场景，防止超卖、重复借阅等并发问题，确保数据一致性和业务正确性。

技术栈：Python 3.12+ + FastAPI + SQLAlchemy 2.0 + PostgreSQL + Redis

---

## 3. 数据库事务策略

### 3.1 事务隔离级别

**设计思路**: 默认使用 PostgreSQL 的 `READ COMMITTED` 隔离级别。对于关键操作（如借书），通过 `get_high_isolation_db()` 提供可设置 `SERIALIZABLE` 隔离级别的会话，防止幻读。异步引擎配置 `pool_pre_ping=True` 自动检测断连。

**会话管理**:
- `get_db()`: 标准会话，自动管理 commit/rollback
- `get_high_isolation_db()`: 高隔离级别会话，通过 `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE` 设置

> 实现文件: `app/db/session.py`

### 3.2 事务传播策略

**设计思路**: `TransactionManager` 提供事务上下文管理器，支持嵌套事务检测（避免重复 commit）。`nested_transaction()` 使用 `SAVEPOINT` 实现部分回滚能力，适用于复杂业务场景中某个子操作失败但整体事务需要继续的情况。

**使用规范**:
1. 简单场景: 通过 `Depends(get_db)` 自动管理事务
2. 复杂场景: 使用 `TransactionManager.transaction()` 显式控制
3. 部分回滚: 使用 `nested_transaction()` 实现 SAVEPOINT
4. 只读优化: `SET TRANSACTION READ ONLY` 提升查询性能

> 实现文件: `app/core/transaction.py`

## 4. 行级锁（SELECT FOR UPDATE）

### 4.1 行级锁原理

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          行级锁工作原理                                      │
└─────────────────────────────────────────────────────────────────────────────┘

无锁情况:
┌─────────────┐              ┌─────────────┐
│  事务A      │              │  事务B      │
├─────────────┤              ├─────────────┤
│ SELECT stock│              │ SELECT stock│
│ stock=10    │              │ stock=10    │
│             │              │             │
│ UPDATE      │              │ UPDATE      │
│ stock=9     │              │ stock=9     │
│ COMMIT      │              │ COMMIT      │
└─────────────┘              └─────────────┘

结果: stock=9 (应该是8，更新丢失!)


有锁情况:
┌─────────────┐              ┌─────────────┐
│  事务A      │              │  事务B      │
├─────────────┤              ├─────────────┤
│ SELECT ...  │              │             │
│ FOR UPDATE  │─────────────►│ 等待...     │
│ stock=10    │              │             │
│             │              │             │
│ UPDATE      │              │             │
│ stock=9     │              │             │
│ COMMIT      │─────────────►│ SELECT ...  │
│ (锁释放)     │              │ FOR UPDATE  │
│             │              │ stock=9     │
│             │              │ UPDATE      │
│             │              │ stock=8     │
│             │              │ COMMIT      │
└─────────────┘              └─────────────┘

结果: stock=8 (正确!)
```

### 4.2 行级锁实现

**设计思路**: `InventoryService` 使用 `select(Book).where(...).with_for_update()` 对图书行加锁，确保同一时刻只有一个事务能修改库存。批量操作时按 `book_id` 排序加锁，防止死锁。

**核心方法**:
- `decrease_stock(book_id, quantity)`: 加锁读取 -> 检查库存 -> 扣减 -> flush
- `increase_stock(book_id, quantity)`: 加锁读取 -> 检查不超过总数 -> 增加 -> flush
- `batch_decrease_stock(book_quantities)`: 按 book_id 排序后依次加锁扣减

**借书流程** (`BorrowService.borrow_book`):
1. 检查用户状态（不需要锁）
2. 依赖数据库唯一索引防止重复借阅: `UNIQUE(user_id, book_id) WHERE status IN ('borrowed', 'overdue')`
3. 调用 `InventoryService.decrease_stock()` 扣减库存（带行级锁）
4. 创建借阅记录

> 实现文件: `app/services/inventory_service.py`, `app/services/borrow_service.py`

### 4.3 锁等待超时处理

**设计思路**: 通过 `SET LOCAL lock_timeout` 和 `SET LOCAL statement_timeout` 配置锁等待和语句执行超时。默认锁等待 5 秒，语句超时 10 秒。超时后向用户返回友好提示"系统繁忙，请稍后重试"。

> 实现文件: `app/core/lock_config.py`

## 5. 乐观锁（版本号）

### 5.1 乐观锁原理

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          乐观锁工作原理                                      │
└─────────────────────────────────────────────────────────────────────────────┘

数据库表结构:
┌─────────┬──────────┬─────────┬─────────┐
│   id    │  title   │  stock  │ version │
├─────────┼──────────┼─────────┼─────────┤
│  book_1 │ Python   │   10    │    1    │
└─────────┴──────────┴─────────┴─────────┘


并发更新场景:

时间线:
  T1          T2          T3          T4
  │           │           │           │
  ▼           ▼           ▼           ▼
事务A读取 ──► 事务B读取 ──► 事务A更新 ──► 事务B更新
stock=10    stock=10     WHERE       WHERE
version=1   version=1    version=1   version=1
                          stock=9     stock=9
                          version=2   
                          
                          成功!        失败!
                                    (version已变为2)


SQL示例:
-- 事务A
UPDATE books 
SET stock = 9, version = 2 
WHERE id = 'book_1' AND version = 1;
-- 影响行数: 1

-- 事务B
UPDATE books 
SET stock = 9, version = 2 
WHERE id = 'book_1' AND version = 1;
-- 影响行数: 0 (version已改变)
```

### 5.2 乐观锁实现

**设计思路**: 通过 `VersionedMixin` 为模型添加 `version` 字段，SQLAlchemy 的 `version_id_col` 配置自动处理乐观锁。`BookServiceWithOptimisticLock` 在更新时捕获 `StaleDataError`，支持指数退避重试（默认最多 3 次）。

**核心类**:
- `VersionedMixin`: 提供 `version` 字段和 `increment_version()` 方法
- `Book(Base, VersionedMixin)`: 配置 `__mapper_args__ = {"version_id_col": version}`
- `BookServiceWithOptimisticLock.update_book_stock()`: 读取 -> 校验 -> 更新 -> 捕获冲突重试

**手动乐观锁**: 也可通过 `UPDATE ... WHERE version = ?` 手动实现，检查 `rowcount > 0` 判断是否成功。

> 实现文件: `app/models/base.py`, `app/services/book_service_with_optimistic_lock.py`

### 5.3 乐观锁 vs 悲观锁选择

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 借书扣库存 | 悲观锁 | 冲突概率高，需要强一致性 |
| 还书加库存 | 悲观锁 | 冲突概率高 |
| 图书信息更新 | 乐观锁 | 冲突概率低，性能好 |
| 用户信息更新 | 乐观锁 | 冲突概率低 |
| 罚款缴纳 | 悲观锁 | 涉及金额，必须强一致 |
| 统计报表 | 无锁 | 只读操作，允许脏读 |

## 6. 幂等性设计

### 6.1 幂等性原理

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          幂等性设计原理                                      │
└─────────────────────────────────────────────────────────────────────────────┘

非幂等操作的问题:

用户点击"缴纳罚款"按钮
       │
       ├───► 请求发送到服务器
       │         │
       │         ▼
       │    服务器处理成功
       │    扣款完成
       │         │
       │         ▼
       │    网络超时(响应丢失)
       │
       ├───► 用户重试
       │         │
       │         ▼
       │    服务器再次处理
       │    重复扣款! ❌
       


幂等操作的设计:

用户点击"缴纳罚款"按钮
       │
       ├───► 请求发送到服务器
       │     携带幂等键: idempotency_key=uuid-123
       │         │
       │         ▼
       │    检查幂等键是否存在
       │         │
       │    ┌────┴────┐
       │    │ 存在     │ 不存在
       │    ▼          ▼
       │  返回缓存    执行扣款
       │  的结果      保存幂等键
       │              和结果
       │                 │
       │                 ▼
       │            网络超时
       │
       ├───► 用户重试
       │     携带相同幂等键
       │         │
       │         ▼
       │    检查幂等键
       │    已存在!
       │         │
       │         ▼
       │    返回缓存的结果
       │    不会重复扣款 ✓
```

### 6.2 幂等键实现

**设计思路**: `IdempotencyKeyManager` 基于 Redis 实现幂等键管理:
- **幂等键生成**: `SHA256(user_id:operation:payload_sorted_json)` 或客户端通过 `X-Idempotency-Key` 请求头传入
- **处理流程**: 检查 key 是否存在 -> 不存在则存储占位符(status=processing) -> 执行业务 -> 存储结果(status=completed) -> 失败则释放 key 允许重试
- **冲突处理**: 如果 key 已存在且状态为 processing，返回 409 Conflict；状态为 completed，直接返回缓存结果
- **TTL**: 默认 24 小时

**幂等装饰器**: `@idempotent(operation="fine_payment", ttl=3600)` 可直接装饰 API 端点，自动处理幂等逻辑。

> 实现文件: `app/core/idempotency.py`

### 6.3 幂等性API使用

**关键端点**:
- `POST /fines/{fine_id}/pay` -- 使用 `@idempotent` 装饰器，自动幂等
- `POST /fines/batch-pay` -- 手动幂等控制，生成包含 `fine_ids` 排序后的幂等键

> 实现文件: `app/api/v1/endpoints/fines.py`

## 7. 库存原子更新

### 7.1 数据库原子操作

**设计思路**: `AtomicInventoryService` 使用 `UPDATE ... SET available_copies = available_copies - ? WHERE available_copies >= ?` 实现无需显式加锁的原子操作。通过检查 `rowcount > 0` 判断是否更新成功（库存不足时 rowcount 为 0）。

**核心方法**:
- `atomic_decrease(book_id, quantity)`: 原子扣减，WHERE 条件包含 `available_copies >= quantity`
- `atomic_increase(book_id, quantity)`: 原子增加，WHERE 条件包含 `available_copies + quantity <= total_copies`
- `atomic_batch_decrease(items)`: 按 book_id 排序后依次原子扣减

**UPSERT 方案**: 对于库存表和图书表分离的场景，使用 PostgreSQL 的 `INSERT ... ON CONFLICT ... DO UPDATE` 实现原子库存更新。

> 实现文件: `app/services/atomic_inventory.py`

### 7.2 库存预扣与释放

**设计思路**: `ReservationService` 实现库存预扣机制，用于下单前的库存锁定:
- **预扣**: 原子扣减库存 -> 创建预扣记录（TTL 5分钟）
- **确认**: 将预扣记录标记为 CONFIRMED，库存已扣减无需再操作
- **取消**: 恢复库存 -> 标记预扣记录为 CANCELLED
- **过期清理**: 定时任务扫描过期预扣记录，自动恢复库存

> 实现文件: `app/services/reservation_service.py`

## 8. 分布式锁（Redis）

### 8.1 分布式锁原理

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          分布式锁原理                                        │
└─────────────────────────────────────────────────────────────────────────────┘

场景: 多个服务实例同时处理同一本图书的借阅

无分布式锁:
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│  服务A      │         │   Redis     │         │  服务B      │
├─────────────┤         ├─────────────┤         ├─────────────┤
│ 读取库存=1   │         │             │         │ 读取库存=1   │
│             │         │             │         │             │
│ 扣减库存     │         │             │         │ 扣减库存     │
│ 库存=0       │         │             │         │ 库存=-1 ❌   │
└─────────────┘         └─────────────┘         └─────────────┘


有分布式锁:
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│  服务A      │         │   Redis     │         │  服务B      │
├─────────────┤         ├─────────────┤         ├─────────────┤
│ SET lock    │────────►│ lock:book_1 │         │             │
│ book_1      │         │ = service_a │         │             │
│ NX EX 10    │         │ (10秒过期)   │         │             │
│             │         │             │         │             │
│ 获取锁成功   │         │             │         │ SET lock    │
│             │         │             │◄────────│ book_1      │
│ 读取库存=1   │         │             │         │             │
│             │         │             │         │ 获取锁失败   │
│ 扣减库存     │         │             │         │ 等待或重试   │
│ 库存=0       │         │             │         │             │
│             │         │             │         │             │
│ DEL lock    │────────►│ lock:book_1 │         │             │
│ book_1      │         │ 被删除       │         │             │
│             │         │             │         │             │
│             │         │ lock:book_1 │◄────────│ SET lock    │
│             │         │ = service_b │         │ book_1      │
│             │         │             │         │ 获取锁成功   │
│             │         │             │         │ 读取库存=0   │
│             │         │             │         │ 库存不足!   │
└─────────────┘         └─────────────┘         └─────────────┘
```

### 8.2 Redis分布式锁实现

**设计思路**: `DistributedLock` 使用 `SET key value NX EX seconds` 原子操作获取锁，使用 Lua 脚本保证释放锁的原子性（只有锁值匹配时才删除）。支持锁续期（`extend`）和上下文管理器（`async with`）。

**核心参数**:
- `lock_key`: 格式为 `lock:{resource_key}`
- `lock_value`: UUID，防止误删其他客户端的锁
- `expire_seconds`: 默认 30 秒，防止死锁
- `retry_times`: 默认 3 次，指数退避重试

**LockManager**: 全局锁管理器，提供 `lock()` 和 `acquire_lock()` 便捷方法。

> 实现文件: `app/core/distributed_lock.py`

### 8.3 分布式锁使用

**设计思路**: 借书时使用 `lock_manager.acquire_lock(f"book:{book_id}")` 获取分布式锁，在锁保护下执行行级锁查询和库存扣减，实现分布式锁 + 数据库锁双重保护。

**批量操作防死锁**: 按 `book_id` 排序后依次获取所有锁，获取失败时释放已获取的锁，确保所有请求以相同顺序加锁。

> 实现文件: `app/services/borrow_service_with_lock.py`

### 8.4 Redlock算法（多Redis实例）

**设计思路**: `Redlock` 算法在多个独立 Redis 实例上获取锁，需要获得多数派（N/2 + 1）才算成功。考虑时钟漂移（`CLOCK_DRIFT_FACTOR = 0.01`），计算锁的有效时间。适用于对可靠性要求更高的场景。

**流程**: 在所有实例上尝试获取锁 -> 检查获得锁的数量是否达到多数派 -> 计算锁的有效时间（TTL - 获取耗时 - 时钟漂移） -> 有效则返回锁信息，否则释放所有已获取的锁并重试。

> 实现文件: `app/core/redlock.py`

## 9. 并发测试方案

### 9.1 测试工具与策略

**设计思路**: `ConcurrentTester` 工具类封装并发执行逻辑，支持指定并发数，统计成功/失败次数和错误信息。

**测试覆盖**:
- **防超卖测试**: 库存 1 本，10 人同时借，断言只有 1 人成功
- **防重复借阅测试**: 同一用户同时发起 5 次借同一本书，断言只有 1 次成功
- **并发还书测试**: 5 人同时还书，验证库存正确增加到初始值
- **压力测试**: 大量并发借还操作下的库存一致性
- **乐观锁冲突测试**: 15 次并发更新（库存 10），验证最终库存正确
- **分布式锁测试**: 10 人竞争 1 本书，验证只有 1 人成功
- **幂等性测试**: 5 次相同幂等键的请求，验证只有 1 次成功

### 9.2 性能基准

| 指标 | 目标值 |
|------|--------|
| 借书 QPS | 100+ 并发请求/秒 |
| 平均响应时间 | < 500ms |
| 锁竞争完成时间 | < 5 秒（50人竞争1本书） |

> 实现文件: `tests/test_concurrent_borrow.py`, `tests/test_performance.py`

## 10. 监控与告警

**设计思路**: 通过 Prometheus 指标监控并发安全相关数据:

| 指标名 | 类型 | 说明 |
|--------|------|------|
| `concurrent_requests_total` | Gauge | 当前并发请求数（按 endpoint） |
| `lock_wait_seconds` | Histogram | 锁等待时间（按锁类型和资源） |
| `deadlock_total` | Counter | 死锁次数（按资源类型） |
| `optimistic_lock_conflict_total` | Counter | 乐观锁冲突次数（按表名） |
| `inventory_violation_total` | Counter | 库存异常次数（按异常类型） |

提供 `measure_lock_wait` 和 `track_concurrent` 装饰器，方便在业务代码中埋点。

> 实现文件: `app/core/concurrency_metrics.py`
