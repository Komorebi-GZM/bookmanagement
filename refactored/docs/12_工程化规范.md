## 1. Python 3.12+ 代码规范

### 1.1 基础规范

#### 代码格式

- 使用4空格缩进，行长不超过88字符
- 遵循 PEP 8 规范，由 `black` 自动格式化
- 字符串引号统一使用双引号

#### 导入规范

按三组顺序导入，组间空一行，组内按字母排序:
1. **标准库**: `datetime`, `typing`, `contextlib`
2. **第三方库**: `fastapi`, `sqlalchemy`, `pydantic`
3. **本地模块**: `core.exceptions`, `models.book`, `services.book_service`

#### 空行规范

- 顶级定义之间2空行
- 类内方法之间1空行

### 1.2 Python 3.12 新特性使用

| 特性 | 语法 | 说明 |
|------|------|------|
| 类型参数 (PEP 695) | `class PaginatedResponse[T]:` | 替代 `Generic[T]` |
| 类型别名 (PEP 695) | `type JsonDict = dict[str, any]` | 替代 `TypeAlias` |
| 联合类型 (3.10+) | `value: int \| str` | 替代 `Union[int, str]` |
| 内置泛型 (3.9+) | `list[int]`, `dict[str, int]` | 替代 `List[int]`, `Dict[str, int]` |
| f-string调试 | `f"{user_id=}"` | 输出 `user_id=123` |
| match-case (3.10+) | `match error: case BookNotFoundError():` | 模式匹配 |
| walrus操作符 | `if book := await service.get(isbn):` | 赋值表达式 |

### 1.3 异步编程规范

- 所有IO操作使用 `async/await`，禁止在异步函数中使用同步IO
- 文件操作使用 `aiofiles`，不使用内置 `open()`
- 并行独立操作使用 `asyncio.gather()`
- 数据库操作使用 `async with db.transaction()` 确保事务一致性

### 1.4 错误处理规范

- 自定义异常继承 `BaseAppException`，提供 `message`、`code`、`status_code`
- 异常捕获要有针对性，禁止裸 `except:` 或 `except Exception: pass`
- 使用上下文管理器 (`asynccontextmanager`) 管理数据库会话等资源
- 异常链使用 `raise ... from e` 保留原始异常信息

## 2. 命名规范

### 2.1 命名约定表

| 类型 | 命名方式 | 示例 | 说明 |
|------|----------|------|------|
| **模块** | 小写 + 下划线 | `book_service.py` | 简短、有意义 |
| **包** | 小写 + 下划线 | `api/routes/` | 避免与标准库冲突 |
| **类** | 大驼峰 | `BookService` | 名词或名词短语 |
| **异常** | 大驼峰 + Error | `BookNotFoundError` | 继承自 Exception |
| **函数** | 小写 + 下划线 | `get_book_by_id` | 动词或动词短语 |
| **方法** | 小写 + 下划线 | `validate_isbn` | 动词或动词短语 |
| **变量** | 小写 + 下划线 | `book_count` | 有意义的名字 |
| **常量** | 大写 + 下划线 | `MAX_BORROW_DAYS` | 模块级别 |
| **私有** | 单下划线前缀 | `_internal_cache` | 内部使用 |
| **强私有** | 双下划线前缀 | `__private_attr` | 避免子类冲突 |
| **类型变量** | 大驼峰/单字母 | `T`, `ResponseT` | 泛型参数 |

### 2.2 布尔变量命名

- 使用肯定语气，以 `is/has/can/should` 开头
- 示例: `is_active`, `has_permission`, `can_borrow`, `should_notify`
- 避免: `not_found`(模糊), `no_error`(双重否定), `flag`(无意义)

### 2.3 集合类型命名

- 使用复数或明确的后缀
- 示例: `books: list[Book]`, `user_ids: set[int]`, `book_by_id: dict[int, Book]`
- 避免: `book_list`(冗余), `user_id_list`(应使用 `user_ids`)

## 3. 类型注解规范

### 3.1 基础规则

- 函数参数和返回值**必须**注解
- 使用 Python 3.10+ 联合类型语法 `int | str`，不使用 `Union[int, str]`
- 使用 Python 3.9+ 内置泛型 `list[int]`，不使用 `List[int]`
- 使用 Python 3.12+ 类型别名语法 `type JsonDict = dict[str, any]`

### 3.2 复杂类型注解

- 回调函数: `Callback = Callable[[int], None]`
- 异步回调: `AsyncCallback = Callable[[int], Awaitable[None]]`
- Pydantic模型: 使用 `model_config` 配置示例数据
- 自引用: 使用 `Self` 类型 (Python 3.11+)

### 3.3 类型别名

为复杂类型创建别名，提高可读性:
```python
UserId = int
BookId = int
ISBN = str
Money = Decimal
JsonDict = dict[str, any]
QueryParams = dict[str, str | int | None]
```

### 3.4 类型检查配置

```toml
[tool.mypy]
python_version = "3.12"
warn_return_any = true
disallow_untyped_defs = true
disallow_incomplete_defs = true
ignore_missing_imports = true
strict_equality = true
```

## 4. 文档字符串规范

### 4.1 格式要求

采用 **Google 风格**文档字符串，配合 `ruff pydocstyle` 检查。

**函数文档字符串模板**:
```
"""简短描述(一句话).

详细描述(可选，说明行为和约束).

Args:
    param1: 参数说明
    param2: 参数说明，默认值

Returns:
    返回值说明

Raises:
    ExceptionType: 触发条件

Example:
    >>> function_call()
    expected_result
"""
```

**类文档字符串模板**:
```
"""类的简短描述.

详细描述(职责、使用场景).

Attributes:
    attr1: 属性说明
    attr2: 属性说明
"""
```

### 4.2 模块文档字符串

每个模块顶部包含: 模块功能概述、主要导出、典型用法示例。

### 4.3 API端点文档

使用 FastAPI 装饰器参数提供文档:
- `summary`: 简短摘要
- `description`: 详细描述
- `responses`: 各状态码说明
- `response_model`: 响应模型

## 5. Git提交规范

### 5.1 Conventional Commits 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 5.2 提交类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **feat** | 新功能 | `feat(books): 添加图书搜索功能` |
| **fix** | Bug修复 | `fix(auth): 修复Token过期问题` |
| **docs** | 文档更新 | `docs(api): 更新借阅API文档` |
| **style** | 代码格式 | `style(models): 格式化模型代码` |
| **refactor** | 重构 | `refactor(services): 重构借阅服务` |
| **perf** | 性能优化 | `perf(db): 优化图书查询性能` |
| **test** | 测试相关 | `test(borrows): 添加借阅单元测试` |
| **chore** | 构建/工具 | `chore(deps): 更新依赖版本` |
| **ci** | CI/CD | `ci(github): 添加自动化测试` |
| **revert** | 回滚 | `revert: 回滚图书搜索功能` |

### 5.3 分支管理

```
main (生产分支)
  │
  ├── develop (开发分支)
  │     ├── feature/user-auth (功能分支)
  │     ├── feature/book-search
  │     └── feature/borrow-flow
  ├── release/v1.0.0 (发布分支)
  └── hotfix/security-patch (热修复分支)
```

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 功能分支 | `feature/<描述>` | `feature/user-auth` |
| 修复分支 | `fix/<描述>` | `fix/login-error` |
| 发布分支 | `release/<版本>` | `release/v1.2.0` |
| 热修复分支 | `hotfix/<描述>` | `hotfix/security-patch` |

### 5.4 提交前检查清单

- [ ] 代码通过所有测试
- [ ] 代码符合类型检查 (mypy)
- [ ] 代码通过代码检查 (ruff)
- [ ] 提交信息符合 Conventional Commits
- [ ] 相关文档已更新
- [ ] 没有敏感信息泄露

## 6. 日志规范

### 6.1 结构化日志

使用 JSON 格式输出结构化日志，包含以下字段:
- `timestamp`: UTC ISO格式时间戳
- `level`: 日志级别
- `logger`: 日志记录器名称
- `message`: 日志消息
- `source`: 文件名、行号、函数名
- `request_id` / `user_id`: 请求上下文(可选)
- `duration_ms`: 执行耗时(可选)

### 6.2 日志级别使用

| 级别 | 用途 | 示例场景 |
|------|------|----------|
| **DEBUG** | 调试信息，仅开发环境 | SQL查询、请求参数详情 |
| **INFO** | 正常业务流程 | 用户登录成功、图书借阅成功 |
| **WARNING** | 非致命问题 | 连接池使用率>80%、响应超阈值 |
| **ERROR** | 需要处理的错误 | 数据库查询失败、认证失败 |
| **CRITICAL** | 需立即处理的严重错误 | 数据库连接丢失、磁盘空间不足 |

### 6.3 日志上下文

- 使用 `contextvars.ContextVar` 传递请求级上下文 (`request_id`, `user_id`)
- 使用 `logging.Filter` 自动将上下文注入每条日志
- 使用装饰器记录函数执行时间，包含 `duration_ms` 字段

### 6.4 日志输出示例

```json
{
  "timestamp": "2024-01-15T10:30:00.123456+00:00",
  "level": "INFO",
  "logger": "api.routes.books",
  "message": "图书借阅成功",
  "source": {"file": "books.py", "line": 156, "function": "borrow_book"},
  "request_id": "req-abc123",
  "user_id": 123,
  "duration_ms": 45.2
}
```

## 7. 代码审查清单

### 7.1 审查检查表

**功能正确性**: 代码实现预期功能、边界条件正确、错误处理完善、无逻辑错误

**代码质量**: 符合PEP 8、命名清晰、函数<50行、圈复杂度<10、DRY原则

**类型安全**: 参数/返回值有注解、复杂类型使用别名、通过mypy检查

**测试**: 新功能有单元测试、覆盖边界情况、所有测试通过

**文档**: 公共API有文档字符串、复杂逻辑有注释

**安全**: 无硬编码敏感信息、输入已验证、SQL注入/XSS防护

**性能**: 无N+1查询、适当缓存、无明显性能问题

### 7.2 审查流程

```
开发者自测 → 自动化检查(CI) → 人工审查 → 修改完善 → 审查通过 → 合并
```

1. 开发者自测: 本地测试通过 + ruff/mypy 通过 + 提交PR
2. 自动化检查: CI流水线运行(单元测试、覆盖率、安全扫描)
3. 人工审查: 指定审查者逐行审查
4. 修改完善: 开发者根据意见修改并重新提交
5. 审查通过: 审查者批准，合并到主分支

### 7.3 审查评论级别

| 级别 | 含义 | 是否阻塞合并 |
|------|------|-------------|
| **[NIT]** | 轻微建议，可选修改 | 否 |
| **[QUESTION]** | 疑问，需开发者澄清 | 否 |
| **[SUGGESTION]** | 建议，由开发者决定 | 否 |
| **[ISSUE]** | 问题，需要修改 | 是 |
| **[BLOCKER]** | 阻塞，必须修改 | 是 |

### 7.4 常见代码审查要点

需要关注的问题模式:
- 缺少类型注解
- 魔法数字(应使用枚举或常量)
- 函数过长/职责过多(应拆分)
- 嵌套过深(应使用早返回)
- 空异常捕获 `except: pass`(应捕获具体异常)
