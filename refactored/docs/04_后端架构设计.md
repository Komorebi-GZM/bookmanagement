## 2. 各层职责

### 2.1 API层 (Routers)

**职责定位:**
- 接收HTTP请求，解析请求参数
- 调用Service层处理业务逻辑
- 返回统一格式的HTTP响应
- 处理路由级别的权限校验

**设计原则:**
- 只负责HTTP协议相关处理，不包含业务逻辑
- 参数校验使用Pydantic模型
- 异常统一抛出，由异常处理器处理

**设计要点:**
- 使用 `APIRouter` 定义路由组，通过 `prefix` 和 `tags` 组织
- 通过 `Depends()` 注入 Service 实例和当前用户
- 权限校验通过 `Depends(require_role(Role.ADMIN, Role.LIBRARIAN))` 声明
- 响应模型通过 `response_model` 指定，自动完成序列化

---

### 2.2 Service层 (Services)

**职责定位:**
- 实现核心业务逻辑
- 协调多个Repository完成复杂操作
- 处理事务边界
- 实现业务规则校验

**设计原则:**
- 一个Service对应一个业务领域
- 方法命名体现业务语义
- 事务控制在Service层
- 不直接操作数据库，通过Repository访问

**设计要点:**
- 通过构造函数注入所需的 Repository 和 UnitOfWork
- 业务方法内使用 `async with self.uow:` 管理事务
- 校验逻辑按步骤执行：用户校验 -> 配额校验 -> 图书校验 -> 创建记录 -> 更新库存
- 事务提交前所有操作在同一个数据库会话中完成

---

### 2.3 Repository层 (Repositories)

**职责定位:**
- 封装数据访问逻辑
- 提供CRUD基础操作
- 实现复杂查询方法
- 隔离数据库细节

**设计原则:**
- 每个实体对应一个Repository
- 继承基础Repository获得通用CRUD
- 复杂查询使用SQLAlchemy的Query API
- 返回Model对象，不返回Schema

**设计要点:**
- `BaseRepository[T]` 提供泛型基础实现：`get_by_id`, `get_multi`, `add`, `update`, `delete`
- `get_multi` 支持过滤条件和分页，返回 `(items, total)` 元组
- 子类扩展领域特定查询方法，如 `get_active_by_user`, `count_active_by_user`, `get_overdue_records`

---

### 2.4 Model层 (Models)

**职责定位:**
- 定义数据库表结构
- 声明字段类型和约束
- 定义表间关系
- 实现领域方法

**设计原则:**
- 使用SQLAlchemy 2.0声明式语法 (`Mapped`, `mapped_column`)
- 模型类与数据库表一一对应
- 业务相关方法在模型中实现
- 避免在模型中引入外部依赖

**设计要点:**
- 使用 `UUID(as_uuid=True)` 作为主键，默认 `uuid.uuid4`
- 软删除通过 `is_deleted` + `deleted_at` 字段实现
- 时间字段统一使用 `DateTime(timezone=True)`，默认 UTC
- 关系通过 `relationship()` 声明，指定 `lazy` 加载策略
- 领域方法如 `is_active()`, `can_borrow()`, `soft_delete()`, `is_available()`, `decrease_stock()` 封装在模型中

## 3. 依赖注入设计

### 3.1 FastAPI依赖注入体系

```
┌─────────────────────────────────────────────────────────────────┐
│                      依赖注入体系                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌──────────────┐     ┌──────────────┐     ┌──────────────┐   │
│   │   Router     │────►│   Service    │────►│ Repository   │   │
│   │   (API层)    │     │  (业务层)    │     │  (数据层)    │   │
│   └──────────────┘     └──────────────┘     └──────────────┘   │
│          │                    │                    │           │
│          ▼                    ▼                    ▼           │
│   ┌────────────────────────────────────────────────────────┐  │
│   │              FastAPI Depends 注入                      │  │
│   │   Depends(get_db_session)                              │  │
│   │   Depends(get_current_user)                            │  │
│   │   Depends(require_role(...))                           │  │
│   └────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 核心依赖

- **get_db_session**: 异步数据库会话生成器，自动处理异常回滚和会话关闭
- **get_current_user**: 从 JWT Token 解析用户信息，校验用户存在性和有效性
- **require_role(\*roles)**: 角色权限校验依赖，不匹配时抛出 403
- **require_owner_or_role(user_id, \*roles)**: 资源所有者或指定角色权限校验

### 3.3 依赖注入容器

- 使用简易 `Container` 类管理服务注册和解析
- 支持 `register()` 注册服务、`register_instance()` 注册单例、`resolve()` 解析服务
- Repository 通过容器统一注册，Service 通过 FastAPI Depends 注入

## 4. 中间件设计

### 4.1 中间件架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         请求流程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Request ──► CORS ──► Logging ──► Auth ──► Exception ──► Router │
│                                                                 │
│  Response ◄── CORS ◄── Logging ◄── Auth ◄── Exception ◄── Router│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 中间件说明

| 中间件 | 职责 | 设计要点 |
|--------|------|----------|
| **CORS** | 跨域资源共享 | 配置允许的源、方法、头信息；暴露 X-Request-ID 和 X-Process-Time |
| **Logging** | 请求日志记录 | 生成 Request-ID、记录请求/响应信息、计算处理时间、写入响应头 |
| **Auth** | 认证检查 | 公开路径白名单放行；非公开路径检查 Authorization 头格式 |
| **Exception** | 全局异常处理 | 捕获 BusinessException/HTTPException/Exception，返回统一错误格式 |

### 4.3 中间件注册顺序

```
ExceptionMiddleware (最外层捕获异常)
  └─► LoggingMiddleware (记录请求日志)
       └─► AuthenticationMiddleware (认证检查)
            └─► CORS (跨域处理)
```

注册顺序很重要：异常处理在最外层确保所有异常都能被捕获，日志在认证之前确保所有请求都被记录。

## 5. 事务控制策略

### 5.1 事务管理架构

```
┌─────────────────────────────────────────────────────────────────┐
│                      事务控制策略                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Service层方法                                                 │
│        │                                                        │
│        ▼                                                        │
│   ┌─────────────────────────────────────┐                      │
│   │   async with UnitOfWork() as uow:   │                      │
│   │       # 业务操作1                    │                      │
│   │       await repo1.add(obj1)         │                      │
│   │       # 业务操作2                    │                      │
│   │       await repo2.update(obj2)      │                      │
│   │       # 提交事务                     │                      │
│   │       await uow.commit()            │                      │
│   │   # 自动回滚或关闭                   │                      │
│   └─────────────────────────────────────┘                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 Unit of Work模式

- 使用 `async with` 上下文管理器管理事务生命周期
- 进入时创建/复用数据库会话，退出时自动回滚未提交的事务
- 提供 `commit()`, `rollback()`, `flush()` 方法
- 支持外部传入会话（嵌套场景）或自动创建会话（独立场景）

### 5.3 事务传播策略

| 策略 | 说明 | 使用场景 |
|------|------|----------|
| REQUIRED | 默认策略，加入当前事务或新建 | 大多数业务操作 |
| REQUIRES_NEW | 挂起当前事务，新建独立事务 | 日志记录，需独立提交 |
| NESTED | 嵌套事务，可独立回滚 | 复杂业务中的子操作 |
| READ_ONLY | 只读事务，优化查询 | 纯查询操作 |

## 6. 异常处理机制

### 6.1 异常体系设计

```
┌─────────────────────────────────────────────────────────────────┐
│                       异常类层次结构                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Exception                                                     │
│      │                                                          │
│      ├── HTTPException (FastAPI内置)                           │
│      │                                                          │
│      └── AppException (应用基础异常)                            │
│             │                                                   │
│             ├── BusinessException (业务异常)                    │
│             │      ├── ValidationException (参数校验)           │
│             │      ├── PermissionException (权限不足)           │
│             │      └── ResourceException (资源相关)             │
│             │                                                    │
│             └── InfrastructureException (基础设施异常)          │
│                    ├── DatabaseException (数据库)               │
│                    ├── CacheException (缓存)                    │
│                    └── ExternalServiceException (外部服务)      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 错误码设计

- `ErrorCode` 枚举定义所有业务错误码，每个包含 `(code, message, http_status)` 三元组
- 错误码分段：1000-1999(认证), 2000-2999(用户), 3000-3999(图书), 4000-4999(借阅), 9000-9999(通用)
- `AppException` 携带 `error_code`, `detail`, `extra` 信息

### 6.3 全局异常处理器

| 处理器 | 异常类型 | 处理逻辑 |
|--------|----------|----------|
| `app_exception_handler` | AppException | 返回业务错误码和详情 |
| `validation_exception_handler` | RequestValidationError | 提取字段和错误信息，返回400 |
| `sqlalchemy_exception_handler` | SQLAlchemyError | IntegrityError返回409，其他返回500 |
| `general_exception_handler` | Exception | 记录日志，返回500通用错误 |
