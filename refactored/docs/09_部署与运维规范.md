## 1. Docker容器化方案

### 1.1 镜像分层策略

```
┌─────────────────────────────────────────┐
│  应用层 (Application Layer)              │
│  - 业务代码                              │
│  - 静态资源                              │
├─────────────────────────────────────────┤
│  依赖层 (Dependencies Layer)             │
│  - Python包 (requirements.txt)           │
│  - Node模块 (package.json)               │
├─────────────────────────────────────────┤
│  运行时层 (Runtime Layer)                │
│  - Python 3.12-slim                      │
│  - Node 20-alpine                        │
├─────────────────────────────────────────┤
│  基础层 (Base Layer)                     │
│  - Debian Slim / Alpine                  │
└─────────────────────────────────────────┘
```

### 1.2 后端服务 Dockerfile

**设计思路**: 采用多阶段构建，builder 阶段安装依赖，production 阶段复制依赖和代码。创建非 root 用户运行应用，配置健康检查，使用 uvicorn 启动（4 workers）。

**关键配置**:
- 基础镜像: `python:3.12-slim`
- 非 root 运行: `appuser:appgroup`
- 系统依赖: `libpq5`, `curl`
- 健康检查: `curl -f http://localhost:8000/health`，间隔 30s
- 启动命令: `uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4`

> 文件路径: `backend/Dockerfile`

### 1.3 前端服务 Dockerfile

**设计思路**: 构建阶段使用 `node:20-alpine` 执行 `npm ci` 和 `npm run build`，生产阶段使用 `nginx:alpine` 托管静态文件，挂载自定义 nginx 配置。

> 文件路径: `frontend/Dockerfile`

### 1.4 镜像构建规范

| 规范项 | 要求 |
|--------|------|
| 基础镜像 | 使用官方镜像，优先选择 `-slim` 或 `-alpine` 版本 |
| 镜像标签 | 格式: `{service}:{version}-{git_short_sha}`，如 `backend:v1.2.0-a1b2c3d` |
| 多阶段构建 | 必须分离构建环境和运行环境 |
| 非root运行 | 生产容器禁止以root用户运行 |
| 层缓存优化 | 将不常变动的指令放在Dockerfile前面 |
| 镜像扫描 | 构建后自动执行 Trivy/Clair 安全扫描 |

### 1.5 镜像构建脚本

**设计思路**: `build.sh` 脚本接收版本号参数，自动获取 git SHA，构建并推送前后端镜像到镜像仓库，构建完成后执行 Trivy 安全扫描。

> 文件路径: `scripts/build.sh`

## 2. Docker Compose编排

### 2.1 服务架构

```
┌─────────────────────────────────────────────────────────────┐
│                         Nginx (反向代理)                      │
│                    端口: 80 / 443                            │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│   Frontend    │    │    Backend    │    │   Adminer     │
│   (Vue3 SPA)  │    │   (FastAPI)   │    │  (DB管理)      │
│   端口: 80    │    │   端口: 8000  │    │  端口: 8080   │
└───────────────┘    └───────────────┘    └───────────────┘
                              │
                              ▼
                    ┌───────────────────┐
                    │   PostgreSQL      │
                    │   端口: 5432      │
                    │   数据持久化       │
                    └───────────────────┘
                              │
                              ▼
                    ┌───────────────────┐
                    │     Redis         │
                    │   端口: 6379      │
                    │   缓存/会话        │
                    └───────────────────┘
```

### 2.2 开发环境 docker-compose.yml

**设计思路**: 开发环境包含 5 个服务（db, redis, backend, frontend, adminer），后端和前端使用 volume 挂载源码实现热重载，数据库暴露 5432 端口方便本地调试。

```yaml
# docker-compose.dev.yml - 服务骨架
services:
  db:
    image: postgres:16-alpine
    # 健康检查: pg_isready
    # 持久化: postgres_data_dev
  redis:
    image: redis:7-alpine
    # 持久化: redis_data_dev, AOF模式
  backend:
    build: ./backend
    # 环境变量: DATABASE_URL, REDIS_URL, SECRET_KEY, LOG_LEVEL=DEBUG
    # 端口: 8000, 热重载: uvicorn --reload
    # 依赖: db(healthy), redis(started)
  frontend:
    build: ./frontend
    # 端口: 5173, Vite dev server
  adminer:
    image: adminer:latest
    # 端口: 8080
```

> 完整文件: `docker-compose.dev.yml`

### 2.3 生产环境 docker-compose.yml

**设计思路**: 生产环境增加 Nginx 反向代理和 Fluentd 日志收集，后端和前端各部署 2 个副本，配置资源限制（CPU 1.0/内存 512M），数据库仅本地访问（`127.0.0.1:5432`），配置 PostgreSQL 调优参数。

```yaml
# docker-compose.prod.yml - 服务骨架
services:
  nginx:
    image: nginx:alpine
    # 端口: 80, 443
    # 挂载: nginx.conf, SSL证书, 日志目录
  backend:
    image: ${REGISTRY}/library-backend:${VERSION:-latest}
    # 环境变量: 从 .env.prod 注入
    # 部署: 2副本, CPU限制1.0, 内存512M
    # 日志: json-file, 最大10m, 保留3份
  frontend:
    image: ${REGISTRY}/library-frontend:${VERSION:-latest}
    # 部署: 2副本
  db:
    image: postgres:16-alpine
    # 端口: 仅 127.0.0.1:5432
    # 调优: max_connections=200, shared_buffers=256MB
  redis:
    image: redis:7-alpine
    # 调优: maxmemory=256mb, allkeys-lru
  fluentd:
    image: fluent/fluentd:v1.16-debian
    # 挂载: fluent.conf, nginx日志
```

> 完整文件: `docker-compose.prod.yml`

### 2.4 环境启动命令

```bash
# 开发环境
docker-compose -f docker-compose.dev.yml up -d

# 生产环境 (使用环境变量文件)
docker-compose -f docker-compose.prod.yml --env-file .env.prod up -d

# 查看日志
docker-compose logs -f backend

# 执行数据库迁移
docker-compose exec backend alembic upgrade head

# 备份数据库
docker-compose exec db pg_dump -U library_prod library_prod > backup_$(date +%Y%m%d).sql
```

## 3. 环境配置管理

### 3.1 环境分级

| 环境 | 用途 | 数据 | 访问控制 |
|------|------|------|----------|
| **dev** | 本地开发 | 模拟数据 | 开发者本地 |
| **test** | 自动化测试 | 每次重置 | CI/CD流水线 |
| **staging** | 预发布验证 | 生产数据脱敏 | 团队内部 |
| **prod** | 生产运行 | 真实数据 | 严格受限 |

### 3.2 配置文件结构

```
backend/config/
├── __init__.py
├── base.py           # 基础配置
├── development.py    # 开发环境
├── testing.py        # 测试环境
├── staging.py        # 预发布环境
└── production.py     # 生产环境
```

### 3.3 配置加载设计

**设计思路**: 使用 Pydantic Settings (`BaseSettings`) 定义配置基类，通过 `@lru_cache` 单例模式获取配置实例。根据 `ENVIRONMENT` 环境变量自动选择对应的配置类。

**配置项分类**:
- 应用信息: `APP_NAME`, `APP_VERSION`, `DEBUG`, `ENVIRONMENT`
- 数据库: `DATABASE_URL`, `DATABASE_POOL_SIZE`(prod=50), `DATABASE_MAX_OVERFLOW`
- Redis: `REDIS_URL`, `REDIS_DB`
- 安全: `SECRET_KEY`, `ACCESS_TOKEN_EXPIRE_MINUTES`(30), `REFRESH_TOKEN_EXPIRE_DAYS`(7)
- 日志: `LOG_LEVEL`, `LOG_FORMAT`(dev=text, prod=json)
- 分页: `DEFAULT_PAGE_SIZE`(20), `MAX_PAGE_SIZE`(100)
- 文件上传: `MAX_UPLOAD_SIZE`(10MB), `ALLOWED_EXTENSIONS`(.pdf, .epub, .mobi)

> 实现文件: `backend/config/base.py`

### 3.4 环境变量模板

```bash
# .env.example - 复制为 .env 后填写实际值

# 应用配置
ENVIRONMENT=development
DEBUG=true

# 数据库
DATABASE_URL=postgresql://user:password@localhost:5432/dbname

# Redis
REDIS_URL=redis://localhost:6379/0

# 日志
LOG_LEVEL=INFO
LOG_FORMAT=json

# 外部服务 (可选)
SENTRY_DSN=
EMAIL_SMTP_HOST=
EMAIL_SMTP_PORT=587
EMAIL_USERNAME=
EMAIL_PASSWORD=
```

## 4. 数据库迁移

### 4.1 Alembic 配置

**设计思路**: 使用 Alembic 管理数据库迁移，配置 `post_write_hooks` 自动格式化迁移脚本（black）。数据库连接 URL 从环境变量读取，不硬编码在配置文件中。

> 配置文件: `backend/alembic.ini`

### 4.2 迁移脚本结构

**设计思路**: `alembic/env.py` 支持异步迁移，通过 `async_engine_from_config` 创建异步引擎。导入所有模型确保元数据完整，`target_metadata = Base.metadata`。

> 实现文件: `backend/alembic/env.py`

### 4.3 迁移命令规范

```bash
# 创建新迁移 (自动生成)
alembic revision --autogenerate -m "add_user_email_index"

# 创建空迁移 (手动编写SQL)
alembic revision -m "add_custom_function"

# 升级到最新版本
alembic upgrade head

# 降级一个版本
alembic downgrade -1

# 查看当前版本
alembic current

# 查看历史
alembic history --verbose
```

### 4.4 迁移脚本编写规范

**设计思路**: 每个迁移脚本包含 `upgrade()` 和 `downgrade()` 函数，确保可回滚。使用部分索引（`postgresql_where`）优化唯一约束，添加 COMMENT 注释说明索引用途。

### 4.5 数据库种子数据

**设计思路**: `seed_data.py` 脚本创建初始管理员账号（admin/Admin@123）和默认图书分类（计算机科学、文学、历史、自然科学）。使用异步会话执行，可重复运行。

> 实现文件: `backend/scripts/seed_data.py`

## 5. 日志收集方案

### 5.1 日志架构

```
┌─────────────────────────────────────────────────────────────┐
│                      应用日志产生层                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ Backend  │  │ Frontend │  │  Nginx   │  │ Database │    │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘    │
└───────┼─────────────┼─────────────┼─────────────┼──────────┘
        │             │             │             │
        └─────────────┴─────────────┴─────────────┘
                          │
                          ▼
              ┌─────────────────────┐
              │     Fluentd         │
              │   (日志收集代理)      │
              └──────────┬──────────┘
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
    ┌───────────┐ ┌───────────┐ ┌───────────┐
    │ 本地文件   │ │  Elasticsearch │ │ 云日志服务  │
    │ (开发/测试)│ │  (生产Kibana)  │ │ (阿里云SLS) │
    └───────────┘ └───────────┘ └───────────┘
```

### 5.2 结构化日志配置

**设计思路**: `CustomJsonFormatter` 继承 `pythonjsonlogger.JsonFormatter`，为每条日志添加标准字段（timestamp, level, logger, source）和环境信息（service, environment）。生产环境使用 JSON 格式，开发环境使用文本格式。

**日志字段**:
- `timestamp`: UTC ISO 格式
- `level`: 日志级别
- `logger`: 日志器名称
- `source`: 文件路径、行号、函数名
- `service`: 服务标识
- `environment`: 环境标识

> 实现文件: `backend/core/logging.py`

### 5.3 日志输出示例

```json
{
  "timestamp": "2024-01-15T10:30:00.123456+00:00",
  "level": "INFO",
  "logger": "api.routes.books",
  "message": "图书借阅成功",
  "source": {
    "file": "/app/api/routes/books.py",
    "line": 156,
    "function": "borrow_book"
  },
  "service": "library-backend",
  "environment": "production",
  "user_id": 123,
  "book_id": 456,
  "borrow_id": 789,
  "duration_days": 30
}
```

### 5.4 Fluentd 配置

**设计思路**: Fluentd 使用 `tail` 插件收集 Nginx 访问日志和错误日志，通过 `record_transformer` 添加 hostname 和 tag 信息。开发环境输出到本地文件（按天分割），生产环境可切换为 Elasticsearch。

> 配置文件: `fluentd/fluent.conf`

### 5.5 日志保留策略

| 环境 | 保留时长 | 存储位置 | 压缩 |
|------|----------|----------|------|
| dev | 7天 | 本地磁盘 | 否 |
| test | 3天 | 本地磁盘 | 否 |
| staging | 30天 | 对象存储 | 是 |
| prod | 90天 | 对象存储 + ES | 是 |

## 6. 监控告警

### 6.1 健康检查端点

**设计思路**: 提供三级健康检查:
- `/health`: 基础检查，返回应用状态、版本、环境信息
- `/health/ready`: 就绪检查，验证数据库和 Redis 连接，返回各依赖服务状态和延迟
- `/health/live`: 存活检查，仅验证应用进程存活

> 实现文件: `backend/api/routes/health.py`

### 6.2 Prometheus 指标

**设计思路**: 定义应用级和业务级 Prometheus 指标，通过 `PrometheusMiddleware` 自动收集 HTTP 请求指标。

**指标分类**:

| 指标名 | 类型 | 说明 |
|--------|------|------|
| `http_requests_total` | Counter | HTTP 请求总数（按 method, endpoint, status_code） |
| `http_request_duration_seconds` | Histogram | 请求延迟（按 method, endpoint） |
| `active_borrows` | Gauge | 当前借出图书数量 |
| `overdue_borrows` | Gauge | 逾期借阅数量 |
| `books_total` | Gauge | 图书总数（按 category） |
| `users_total` | Gauge | 用户总数（按 role） |
| `db_connections` | Gauge | 数据库连接数（按 state） |
| `db_query_duration_seconds` | Histogram | 数据库查询延迟（按 operation） |

> 实现文件: `backend/core/metrics.py`

### 6.3 告警规则

**设计思路**: 基于 Prometheus 告警规则，覆盖服务可用性、错误率、延迟、资源使用等维度。

| 告警名称 | 触发条件 | 级别 | 持续时间 |
|----------|----------|------|----------|
| ServiceDown | 后端服务不可用 | critical | 1分钟 |
| HighErrorRate | 5xx 错误率 > 5% | warning | 2分钟 |
| HighLatency | P95 延迟 > 2s | warning | 5分钟 |
| DatabasePoolExhausted | 连接池使用率 > 90% | critical | 1分钟 |
| DiskSpaceLow | 磁盘剩余 < 10% | warning | 5分钟 |
| MemoryUsageHigh | 内存使用 > 85% | warning | 5分钟 |

> 配置文件: `monitoring/alert-rules.yml`

### 6.4 告警通知配置

**设计思路**: Alertmanager 按严重级别路由告警: critical 级别发送邮件给管理员 + Slack 通知，warning 级别仅发送运维邮件。告警分组按 alertname 和 severity，30s 内合并，4h 内不重复。

> 配置文件: `monitoring/alertmanager.yml`

## 7. 备份策略

### 7.1 备份架构

```
┌─────────────────────────────────────────────────────────────┐
│                      备份策略概览                            │
├─────────────────────────────────────────────────────────────┤
│  PostgreSQL         │  应用配置          │  用户上传文件      │
│  ├─ 全量备份(每日)   │  ├─ Git版本控制    │  ├─ 对象存储       │
│  ├─ 增量备份(每小时) │  └─ 环境变量加密   │  └─ 跨区域复制     │
│  └─ WAL归档(实时)   │                   │                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
              ┌─────────────────────────────┐
              │      备份存储层级            │
              ├─────────────────────────────┤
              │  热存储: 最近7天 (SSD)       │
              │  温存储: 7-30天 (标准存储)   │
              │  冷存储: 30-90天 (低频访问)  │
              │  归档: 90天+ ( Glacier )    │
              └─────────────────────────────┘
```

### 7.2 数据库备份脚本

**设计思路**: `backup-db.sh` 使用 `pg_dump --format=custom --compress=9` 执行全量备份，通过 `pg_restore --list` 验证备份完整性，自动上传到 S3（STANDARD_IA 存储类），清理 7 天前的本地备份和 30 天前的 S3 备份。

> 文件路径: `scripts/backup-db.sh`

### 7.3 数据库恢复脚本

**设计思路**: `restore-db.sh` 支持 S3 路径自动下载，恢复前需二次确认（会 DROP 当前数据库），自动终止现有连接、删除并重建数据库、使用 `--jobs=4` 并行恢复。

> 文件路径: `scripts/restore-db.sh`

### 7.4 定时任务配置

```bash
# Crontab 配置

# 数据库全量备份 - 每天凌晨2点
0 2 * * * /opt/library/scripts/backup-db.sh >> /var/log/library/backup.log 2>&1

# 应用配置备份 - 每周日
0 3 * * 0 tar -czf /backups/config/config_$(date +\%Y\%m\%d).tar.gz /opt/library/config/

# 清理旧日志 - 每天
0 4 * * * find /var/log/library -name "*.log" -mtime +30 -delete

# 健康检查报告 - 每天8点
0 8 * * * /opt/library/scripts/health-report.sh >> /var/log/library/health.log 2>&1
```

### 7.5 备份验证与演练

| 验证项目 | 频率 | 负责人 | 检查内容 |
|----------|------|--------|----------|
| 备份完整性 | 每日 | 自动化 | 校验文件大小、pg_restore --list |
| 备份可恢复性 | 每周 | DBA | 在测试环境执行恢复 |
| 恢复时间测试 | 每月 | 运维团队 | 记录RTO实际值 |
| 灾难恢复演练 | 每季度 | 全体团队 | 模拟完整故障恢复 |

### 7.6 RTO/RPO 目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **RPO** | < 1小时 | 数据丢失最大容忍时间 (WAL归档) |
| **RTO** | < 30分钟 | 服务恢复最大容忍时间 |
| 备份窗口 | < 2小时 | 每日全量备份完成时间 |
| 恢复验证 | < 1小时 | 备份可恢复性验证时间 |

### B. 故障排查流程

```
1. 检查服务状态
   docker-compose ps

2. 查看错误日志
   docker-compose logs --tail=500 backend

3. 检查资源使用
   docker stats
   df -h
```
